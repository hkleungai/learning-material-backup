Overview

Large-scale software development projects require groups of designers, implementers and support staff. Because of this, these projects demand not only technical abilities, but also skills in group dynamics and communication. To give you some experience with all of these aspects of software development, you will work in three person groups to carry out the required project activities including requirement analysis, architecture, design, implementation, and documentation.

The project will be developed in a team of four students. The team formation is a combination of voluntary teaming and random assignment. This means that for teams of insufficient members beyond Feb 17 is subject to random regrouping with the best effort to preserve the existing partnership. In case that the total number of students is not a multiple of four, teams of five will be established by random assigning additional members to the existing groups.

The project will be developed using the Java programming language and the Eclipse Java development toolkit. These development tools will be introduced in lectures, labs, and tutorials. There are tutorial notes that describe the features and functionality of these development tools, which you should read before your lab and tutorial each week! Because the lab and tutorial time is limited, we can cover only some aspects of these development tools. Therefore, you will be expected to learn how to use these development tools mainly on your own.

Your group may lose a member for various reasons. If this happens, you will be expected to close ranks and adjust to the problem. Loss or change of personnel does happen in the real world and is yet another reason for carefully documenting each step of a software development project.

Computer resources seem to be the scarcest and least available when needed most. This is also true in the real world. Servers may be heavily loaded or unavailable at critical times. Do not leave working on a project activity until the last week, last few days or last few hours before a deadline. If you do so, you are just asking for trouble. You have been warned!


Project Organization

For the course project, you are given the source code of a skeleton package that only acts as a starting point for the graphical user interface (GUI) of the calendar system. The GUI contains the minimum necessary elements for which you can use as a design reference and are free to extend, to modify, and to create new ones, as long as your changes do not affect the testing of the requirements of the project.

The project will be divided into two major phases: the single user memory-based appointment scheduling and the multi-user disk-based scheduling. The learning objectives of the two phases are described as follows.

Phase one:

  • Learn to program in modern object oriented language such as Java using a fully featured integrated development environment (IDE).
  • Learn to convert imprecise textual technical requirements to precise technical specifications.
  • Learn to extend the given code base by adding new GUI elements and major new features.
  • Learn to develop software as a controlled and monitored process through the use of version control.

    Phase two:

  • Learn to support the notion of persistence and data management.
  • Learn to use unit testing and regression testing to maintain the quality of the software.
  • Learn to articulate and document the design and architecture of software systems.

    Evaluation

    The assessment of the project Phase I and II is based on both functional correctness (80%) and the coding style (20%). The functional correctness is measured through two components: group assessment (70%) and individual assessment (30%). In the group assessment, you will be asked to demo your submitted program through a set of test cases. Each test case is a set of operational instructions, conducted by group members, with the expected output, verified by the TA. The group score is determined by the score sheet of the test cases. To assess the individual contribution, each team is required to submit a document describing the following:
    1. Individual contribution percentage: each member of the team needs to specify his or her contribution to the milestone. I would expect each member will contribute 100% of her or his share of the work. Any case that has lower than 70% contribution will lead to a meeting with the TA or the instructor.
    2. Each group member will need to indicate which features of the calendar system that she or he has worked on. The feature must match one of the numbered requirements stated in the problem statement.
    This document will be used in the demo to evaluate the individual contribution by asking the responsible group member to answer code-level questions and to make small modifications of your code on the fly.

    Time machine. Do not program directly using your computer clock because, for testing purposes, we will ask you to fast forward and rewind the clock that is used by the calendar system. During the evaluation of the project, the evaluator will use the interface you provide, either on the GUI or from the console, to use this time travelling feature and test the functionalities of the implementation. You must make sure your entire calendar system is based on a changable clock.

    Please make note of the following:
    1. You will need to self-educate with the most of the related material. Google is your best friend.
    2. You must discuss the design and the technical details in the architectural document to obtain the full mark.
    3. You are only required to implement these specifications in the final submission of your project.

    Architecture Document

    For the system design document, you are required to write a design document that describes the use of design patterns in your project. You are required to identify the usages of two design patterns in the existing code and one new application of design patterns that can improve the existing design of your code. For the new application, you are not required to implement them in your code, since this is a design document. For each of the three descriptions of patterns, you should follow the following sectioning guidelines:
    1. Name: The name of the pattern.
    2. Motivation: What design problem, in the context of the calendar software, exists without the use of this particular pattern. You are encouraged to think in the context of real world usages that involve many users and a large amount of information.
    3. Solution: How this pattern solves the problem for the calendar software in theory according to the real world usage.
    4. Class diagram: Give the class diagram in each description that shows the proper inheritance relationships of Calendar classes with respect to the general pattern classes, in the style consistent with the lecture notes. For instance, you should diagram the generic classes of Observer pattern, and show how your classes inherit from them. You are not required to show all fields and methods for the clarity of the diagram. You can just show the necessary fields and methods that are relevant to the implementation of the specific pattern.
    The requirement for the document is a A4/12 point font for no less than 3 pages. The document will be graded on the correctness and clarity of the presentation and the discussion.

    Bonus Competition (20% of the project mark)

    To encourage creativity and have fun, there is a bonus competition that each team can freely implement an unspecified feature that is relevant to the calendar application. To be clear, a feature means a new way for the end user to interact with the calendar system that is useful. This bonus feature will be evaluated by the teaching team at the end of the sememster. The top 3 teams will be rewarded the full project bonus (20%) and the next 3 teams will get half of the project bonus (10%). To enter the bonus competition, you will need to first inform your mentor and, most importantly, submit a README file to explain how your feature functions. Naturally, you will also need to indicate the team members' responsibilities. The bonus feature will be checked in the demo based on your README document.
