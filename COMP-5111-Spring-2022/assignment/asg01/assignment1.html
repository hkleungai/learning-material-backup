Assignment Tasks

Task 1: Test case generation and Code coverage measurement (30%)

You are expected to finish 2 sub tasks.

Task 1.1: To learn how to use Randoop for test cases generation.

A short tutorial about Randoop is here, including how to use Randoop to generate test case, as well as run the test case in Eclipse.

You are expected to use Randoop to generate 5 different test suits for the program under test.

Each test suit should have statement coverage more than 50%. (In EclEmma, please refer to the percentage value of "Utils.java" using "instruction coverage")

You should use the parameter --randomseed to control the random seed, otherwise Randoop may generate the same test in different runs.

The generation should use the parameter --junit-package=asg.

The 1st run should use the parameter--junit-output-dir=testUtils0, and the 2nd run should use the parameter --junit-output-dir=testUtils1... until --junit-output-dir=testUtils4

Submissions : the 5 test suits generated by you, i.e., the folder testUtils0, testUtils1, testUtils2 testUtils3, testUtils4.

Grading Scheme: Each test suit is for 3% if it can be successfully executed.

Task 1.2: To learn how to use EclEmma for code coverage measurement.

EclEmma is a Eclipse plugin to measure the code coverage. A tutorial is here here.

You are expected to use EclEmma measure the coverage of the 5 test suits submitted by you.

Submissions: For each suit, you should submit a screenshot showing the instruction coverage (line coverage is also acceptable) and a screenshot for branch coverage. In total, you are expected to submit a folder containing 10 screenshots. Each screenshot should be properly renamed so one can know which suit it is for, and whether it is for line coverage or branch coverage.

Grading Scheme: Each screenshot is for 1.5% if the values in the screenshot are correct.

Task 2: Statement coverage measurement using Soot (40%)

Soot is a tool for java program optimization. In our assignment, we use it for code instrument, so that we can measure the code coverage.

We prepare a tutorial about Soot, here. Some common questions are available, here.

You are expected to use Soot to measure the statement coverage. You should use Soot to insert some special code after each line of bytecode. When the test cases are running, these special codes will be executed if and only if the associated bytecode is executed, so that they can record the coverage. After all, your tool should be able to report the statement coverage achieved by the generated test suite.

To validate the correctness of your statement coverage report, you can compare it with the report generated by EclEmma.

Note:Since we use soot analyzes the Jimple code, the statement here actually refers to Jimple statement, instead of Java statement. EclEmma uses java bytecode statement, so it is normal that their results are not same

Please put your code under the folder src/coverage

The entry point should be src/coverage/MainDriver.java, which we have created a template for you. You can add new class under this package.

Program output:

Your program should at least output the following things:

  1. Each statement, and whether it is covered (yes or no).
  2. The statement coverage for whole class (percentage)

Submissions:

  1. You are required to provide your source code.
  2. You are also required to provide some script for execution, like ant, maven, or makefile.
  3. A readme file is required to clearly state how to run your code.
  4. A file showing the difference of the percentage (coverage for whole class) report by your tool and EclEmma. Minor difference is allowed. Your should use all the 5 test suits generated by you.

Grading Scheme: 30% if your program can successfully instrument the code and report the coverage. 10% if the results by your code are correct. Your mark will be deducted if there is no readme or running script. Your mark will be deducted if you program can be executed but the output is incorrect.
