Test Coverage Report Generation with Soot

Introduction

In this project, we will use the tool Soot to instrument our subject program and generate the statement coverage report. Soot is a widely-used Java optimization framework. Soot can be used as a stand alone tool to optimize or inspect class files, as well as a framework to develop optimizations or transformations on Java bytecode. For more detailed introduction to Soot, you can refer to the website https://sable.github.io/soot/.

A reference for soot is here. This document is based on old version of soot, however, it can help you understand a lot of basic concept and usage of soot.

 

In this project, you are required to instrument our subject programs and your generated test cases. Then, after executing the instrumented programs, the statement coverage report will be generated based on the execution. To evaluate the project, you can compare your generated test coverage reports with those generated by the existing tool EclEmma.

Steps

You need to follow these steps to use Soot to generate test coverage report.

1. Set JAVA_HOME

Before running soot, please make sure that environment variable JAVA_HOME is correctly set on your computer (especially MacOS users).

You may check by the following command:

Linux and MacOS (bash):

echo $JAVA_HOME

Windows:

echo %JAVA_HOME%

If the command shows nothing, please set your JAVA_HOME variable to your path to your installed jdk

If JAVA_HOME is not properly configured, soot may throw an exception that asks you to add “rt.jar” to your soot-classpath.

2. Download Soot

Please download Soot from here.

You are also recommended to use the version included in our example soot project that is available in our Soot workshop.

3. Read the Soot documentations on how to instrument java programs using soot

Soot is a complex tool and the learning path we recommended is to learn it by examples. When you read the examples, if you are not clear about anything, you can search it in GitHub Soot wiki

You can check the following tutorials on how to instrument java programs using Soot. The source code is also provided in the tutorials. https://github.com/Sable/soot/wiki/Adding-profiling-instructions-to-applications

The example in our workshop shows how to instrument java program using Soot in order to count how many times static methods are called. The comments in source code almost explain every line of code in details.

4. Implement your own tool

To generate the test coverage reports, you need to conduct the following steps:

· Analyze and instrument the Java programs. In this step, you should consider which part of the programs you are going to instrument, and what kind of information you are going to put in the program instrumentation.

· Analyze the execution log and generate the test coverage report. After instrumentation, you need to run the test cases and the the execution log will be generated then. Then, based on the execution log, you can compute the test coverage.

 

Please put your code under the folder src/coverage

The entry point should be src/coverage/MainDriver.java, which we have created a template for you. You can add new class under this package.

5. Cross validation

You can compare the test coverage report generated by your tool and the report generated by EclEmma for cross validation.
