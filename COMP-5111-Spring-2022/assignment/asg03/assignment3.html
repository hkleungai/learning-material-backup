COMP5111 (Spring 2020) Course Assignment 3

Deadline: 31st May 23:55

Assignment Objectives

The objectives of this assignment are to (1) practice dataflow coverage using Soot and (2) leverage it to enhance the fault localization in assignment 2. It consists of four required tasks and one bonus task:

  1. Find Def-Use Pairs (10%)
  2. Find Def-Use Paths (20%)
  3. Measure All-Def-Use-Path Coverage (30%)
  4. Leverage All-Def-Use-Path Coverage to enhance the fault localization in assignment 2 (40%)
  5. (Bonus Task) Further enhance the Fault Localization using test requirements other than lines/statements, branches and def-use paths. (20%)

Assignment Tasks

Please complete the following tasks using Soot.

Task 1: Find Def-Use Pairs (10%).

In this task, please use Soot to find the Def-Use Pairs for each variable in each method.

To get each variable in each method, you can use the API getLocals.
For each statement you can use the API getDefBoxes and getUseBoxes to know which variable is defined and used in this statement.

Your program should output two files:

  1. Mapping file. In this file, each line is in the format of “method signature[TAB]jimple statement[TAB]index”. The “index” refers to the index of this statement in this method body “body.getUnits()”. Instrumented statements should not be indexed. Please name this file as mapping_du-pairs.csv.
  2. Def-Use Pair. In this file, each variable should have two lines: the first line should be in the format of “method signature[TAB]variable”. The second line should list the def-use pairs. Each pair should be put into parentheses and two pairs should be separated by single tab . An example is like this: “(0,4)[TAB](0,5)[TAB](2,4)[TAB](2,5)”. The pairs should be sorted according to the ascending order of the 1st index, followed by the ascending order of the 2nd index. In each pair, the number is the of def/use statement. The actual def/use statement should be identified via looking up the index in the above mapping file. For example, if (0,3) is a def-use pair of variable a, it means the variable a is defined in 0th jimple statement of this method body and is used in 3rd jimple statement. Please name this file as du-pairs.csv.

Implementation Requirements:

  1. Your program should have the entry point in dataflowAnalysis.duPairs. When grading, we will delete any existing .class and .csv files. Then we will re-compile your code and run java -cp necessary_classpath dataflowAnalysis.duPairs -p jb use-original-names:true asg.Utils asg.Utils\$IOCase in command line (or eclipse). Your program should output the two files correctly. It is possible that we will use another subject program for grading.

Please provide the scripts, instructions and readme to run your program. Make sure the scripts/instructions/readme explain how to compile your code.
  2. mapping_du-pairs.csv and du-pairs.csv

Grading Scheme:

Marks will be deducted if your code list more def-use pairs, or miss any def-use pairs. Marks will be deducted if your code does not follow the above output format and implementation format.

Task 2: Def-Use Paths (20%).

Reminder: def-use path of variable v is a simple path that is def-clear with respect to v
from a def of v to a use of v

Your program should list all def-use paths for each variable.

Your program should output two files:

  1. Mapping file. In this file, each line is in the format of “method signature[TAB]jimple statement[TAB]index”. The “index” refers to the index of this statement in this method body “body.getUnits()”. The instrumented statements should not be indexed. Please name this file as mapping_du-paths.csv.
  2. Def-Use Path. In this file, each variable should have two lines: the first line should be in the format of “method signature[TAB]variable name”. The second line should list the def-use paths. Each path should be put into brackets and two paths should be separated by a single tab. The paths should be sorted by the ascending order of their length. For the paths having the same length, they should be sorted according to the ascending order of the 1st index, then 2nd index, 3rd, … An example is like this: “[0,1,3,4][TAB][0,2,3,4]”. For each path, the number is the of def/use statement. The actual def/use statement should be identified via looking up the index in the above mapping file. Please name this file as du-paths.csv.

Implementation Requirements:

  1. You program should have an entry point in dataflowAnalysis.duPaths. In the grading, we will delete any existing .class and .csv files. Then we will re-compile your code and run java -cp necessary_classpath dataflowAnalysis.duPaths -p jb use-original-names:true asg.Utils asg.Utils\$IOCase in the command line (or eclipse) and your program should output the two files correctly. It is possible that we will use other subject programs in the grading.

Submissions:

  1. The source code of your program, which should be in the folder comp5111asg03/src/dataflowAnalysis/. You can re-use your code in assignment 1/2. Please provide the scripts, instructions and readme to run your program are required. Make sure your scripts/instructions/readme explain how to compile the code.
  2. mapping_du-paths.csv and du-paths.csv

Grading Scheme:

  1. Correctness of your program (20%): Marks will be deducted if your code list more def-use paths, or miss any def-use paths. Marks will be deducted if your code does not follow the above output format and implementation format.

Task 3: All-Def-Use-Path Coverage (30%).

Your program should measure the All-Def-Use-Path Coverage.

Your program should output three files:

  1. Mapping file. In this file, each line is in the format of “method signature[TAB]jimple statement[TAB]index”. The “index” refer to the index of this statement in this method body “body.getUnits()”. The instrumented statements should not be indexed. Please name this file called mapping_all-du-paths.csv.
  2. All-Def-Use-Path. In this file, each variable should have two lines: the first line should be in the format of “method signature[TAB]variable name”. The second line should list the def-use paths. Each path should be put into brackets and two paths should be separated by a single tab ‘\t’. The paths should be sorted by the ascending order of their length. For the paths having the same length, they should be sorted according to the ascending order of the 1st index, then 2nd index, 3rd, … An example is like this: “[0,1,3,4][TAB][0,2,3,4]”. For each path, the number is the of def/use statement. The actual def/use statement should be identified via looking up the index in the above mapping file. Please name this file called all-du-paths.csv.
  3. Coverage report. For each path of each variable, you should use one line shows whether it is covered or not. The format should be “method signature[TAB]variable name[TAB]path[TAB]YES/NO”. An example is “<void count(int)>[TAB]length[TAB][0,1,3,4][TAB]YES”. At the end of this file, you should report the total number of paths and the covered paths. The format should be “Coverage=covered/total”, e.g., “Coverage=30/40”. Please name this file called coverage_all-du-paths.csv. NOTE: This report will be generated after the test cases are executed with your instrumented program

Implementation Requirements:

  1. You program should have an entry point in dataflowAnalysis.allDuPathsCvg. When grading, we will delete any existing .class and .csv files. Then we will re-compile your code and run java -cp necessary_classpath dataflowAnalysis.allDuPathsCvg -p jb use-original-names:true asg.Utils asg.Utils\$IOCase in the command line (or eclipse). Your program should output the first two files correctly. At last we will run the test cases and your program should output the third files correctly. It is possible that we will use another subject program for grading.

Submissions:

  1. The source code of your program, which should be in the folder comp5111asg03/src/dataflowAnalysis/. You can re-use your code in assignment 1/2. Please provide the scripts/instructions/readme to run your program. Make sure your scripts/instructions/readme explain how to compile the code.
  2. mapping_all-du-paths.csv, all-du-paths.csv and coverage_all-du-paths.csv.

Grading Scheme:

  1. Correctness of your program: List all all-def-use-path (15%). Marks will be deducted if your code list more all-def-use-paths, or miss any all-def-use-paths. Marks will be deducted if your code does not follow the above output format and implementation format.
  2. Correctness of your program: Coverage (15%). Marks will be deducted if your code cannot correctly measure the coverage. Marks will be deducted if your code does not follow the above output format and implementation format.

Task 4: Enhance the Fault Localization using All-Def-Use-Path Coverage (40%).

Now we have the All-Def-Use-Path Coverage and we can leverage this information to enhance the fault localization. Please use the test cases we provided in the assignment 2.

Your task is to design a reasonable methodology to leverage the all-def-use-path coverage and the statement coverage to locate the fault(s) in the subject program.

Note: This is an open question. We expect you regard this task as a small-scale research task. The grading of this section will be not only based on the correctness of your implementation, but also based on the quality of the report, the design of your approach, etc.

  1. If there are any coefficients and parameters in your algorithm, please discuss their purposes and effect.
  2. You should compare the result with the fault localization result in assignment 2.

Your program should output a spectrum report of potential faulty statements. Each spectrum report should in .csv format and each line is in the format of “method signature[TAB]statement[TAB]suspicious score[TAB]ranking”. The report should be sorted according the descending order of suspicious scores. If multiple statements have the same scores, please sort them according to the alphabetical order of method signature and statement. The method signature could be obtained using Soot API getSignature() e.g. body.getMethod().getSignature(). The ranking of suspicious score aa should be computed as (N+M+1)/2, where N is the number of statements whose suspicious scores are higher than aa and M is the number of statements whose suspicious scores are higher than or equal to aa. For example, if a sequence of suspicious scores is (0.9, 0.8, 0.8, 0.7) their rankings are (1, 2, 2, 4), respectively. If a sequence of suspicious scores is (0.9, 0.8, 0.8, 0.8, 0.7, 0.7, 0.6) their rankings are (1, 3, 3, 3, 5, 5, 7), respectively.

  1. The source code of your program, which should be in the folder comp5111asg03/src/dataflowAnalysis/. You can re-use your code in assignment 1/2. Scripts/instructions/readme to run your program are required. Make sure your scripts/instructions/readme explain how to compile the code.
  2. The spectrum reports of potential faulty statements running against the 3 test suits provided by us. In total, you need to submit 3 reports. Please name each of them in the format spectrum_adupc_test[10-12].csv
  3. A written task 4 report, which should at least describe the intuition of your design, the methodology, the experiment setting, the experiment result and discussion. Please name it as task4_report.pdf. Make sure it is an academic report.

Grading Scheme:

  1. Novelty of your approach (10%)
  2. Solidity of your implementation and experiments (20%).
  3. Quality of your report (10%).
