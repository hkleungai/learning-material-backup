COMP2011 Lab 10


Introduction

In this lab, we will use class to design a library management system.
In the library management system, we defined a class called Book to store all the information of a book, including name, number of pages, number of authors and author names. Also, since the books are stored by a linked list, there would a pointer next pointing to the next book.
To simplify, we use a string without space to store the names of books and authors. For example, if a name is "Abc Def Ghi", we use "Abc_Def_Ghi" to represent it.

Tasks

There are 5 tasks in book.cpp. Descriptions of tasks are provided below, instructions and some examples are also provided via comments in the skeleton codes.

  • Task 1: Constructor of a new book.

    • In this part, all the information of a book including numPages, numAuthor, authorList, name, next will be initialized. Note that next would be initialized as nullptr in the constructor, and its value would be set in main.cpp after a book is constructed. Alao, remember to allocate memory for variable name, numAuthor, and each element in the array numAuthor.

  • Task 2: Destructor to delete an existing book.

    • In this part, all the memory allocated previously in the constructor should be deallocated. Note that for an array (e.g., numAuthor), the array as a whole and each element in the array should be deallocated respectively.

  • Task 3: Search for and print out information of book(s) with a given name.

    • Note that the strcmp function can be used to compare two strings. The function printSingleBookInfo() can be used to print out the information of a given book. Since the books are stored in a linked list, next is a pointer to the next book. And for the last book, next should be nullptr.

  • Task 4: Search for and print out information of book(s) of a given author.

    • Note that the strcmp function can be used to compare two strings. The function printSingleBookInfo() can be used to print out the information of a given book. Since the books are stored in a linked list, next is a pointer to the next book. And for the last book, next should be nullptr.

  • Task 5: Search for and delete book(s) of a given name.

    • Note that the strcmp function can be used to compare two strings. Since the books are stored in a linked list, next is a pointer to the next book. And for the last book, next should be nullptr.
    To simplify, in the function deleteBookByName, we would start from the second book. The case that the first book is a book to be deleted are handled in ManagementSystem.cpp .

Grading

Your assignment will be automatically graded by ZINC.

  • In the grading report, pay attention to various errors reported. For example, under the "make" section, if you see a red cross, click on the STDERR tab to see the compilation errors. You must fix those before you can see any program output for the test cases below.
  • Please ensure your program can be compiled without any problem in ZINC. If your program cannot be compiled, you will receive 0 mark.
  • Your implementation will be graded by running six test cases, each contains a different input parameter. A test case should finish within 3 seconds. Failing to do so would result in 0 mark for that test case.
  • Sanitizer checkings will be turned on for test case 1. If it reports any error (including memory leak and other runtime errors detected) you will get 0 for that case. See PA3 page for what that is, and how you may check for memory leak yourself without using ZINC.
  • Your program should not crash during a test case. (e.g. you see a non-zero exit code in your ZINC grading report) If it does, you will receive 0 marks for that test case.
  • Only your latest submission will be graded. You are responsible to make sure the latest submission is correct.
  • Do not hardcode your solution for the given test cases.

About any ZINC server / webpage / UI problem

If you see a blank page or some errors after some action on ZINC, you may try refreshing the page and repeating the action again, and it usually will work the second / third time. If it still doesn't work, try using another browser / private window. Please also email the details about the bug you encountered (potentially with a screenshot if that is helpful) to zinc@cse.ust.hk and they will try to fix the bug as soon as possible.

It is not unusual that reports may not be generated until hours later when deadline approaches. When the ZINC cannot give you the grading report quickly, please test your program on your own computer, and you may compare your output with the sample output we provided yourself. Please do NOT resubmit the same file again and again, as it will lag the server and won’t make it give you the report faster.

While you may not get the grading report quickly, as long as your submission is successful (the grading part and submission collection part are working separately on ZINC, and the submission collection part is usually responsive and stable), we would grade your latest submission with all test cases after the deadline, so don’t worry too much.
