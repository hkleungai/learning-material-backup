General

Q:

 Are acceptance tests just sentences that restate the system requirements in point form?

A:

 The restatement of the requirements is not the acceptance tests. The restatement is useful for writing the acceptance tests. After you have restated the requirements in point form, then you need to decide which of them are actually testable (i.e., can become acceptance tests) and then you need to further devise test cases that will demonstrate that the requirements are implemented correctly.

Q:

 It is suggested that we do a restatement of the requirements when we do the acceptance test cases. Is it necessary for us to hand in the requirements restatement?

A:

It is not necessary for you to hand in the requirements restatement that you do to derive the acceptance test cases. The requirements restatement for deriving the acceptance test cases are just the means to the end, which is deriving the acceptance test cases.

Q: Is the user interface graded in the project?
A: Yes, there is a grade (between 5-10%) for the usability of the user interface of your system that will be included in the Final Acceptance Testing grade. The specific grade will be decided by the person evaluating your system (i.e., it is based on a subjective judgment made by the assessor).

Q: Will you supply the data for testing our system or we have to create our own data for testing.
A: For the testing that we do (i.e., Initial Acceptance Testing and Final Acceptance Testing), we will provide the required data. For Peer Acceptance testing and for the testing that you do (i.e., while you are implementing the system), you should provide the data.

Q:

When our website is tested, must we use only the computers in Lab 4?

A:

Your website must run on the computers in Lab 4. While you can develop your website on any computer you like, for doing acceptance testing you must be able to run your website on the computers in Lab 4.

Important: You should be constantly testing your web application on the computers in Lab 4 to make sure that it will run from there.


Initial Acceptance Testing

Q:

For Initial Acceptance Testing, is the grade proportional to the number of test cases passed?

A:

There is a grade for each acceptance test that needs to be performed and the overall grade is based on how many of the acceptance tests can be satisfactorily passed.


Peer Acceptance Testing

Q:

What do we need to do for Peer Acceptance Testing?

A:

You need to write test cases that will be used to test another team's implementation. The test cases should show that the system works as specified. One way to show this is to construct scenarios of how the system will be used and to demonstrate to the customer through these scenarios that the system works as specified. The scenarios basically come from the use cases, but you need to specify a step-by-step test that the tester can follow in using the system to do them.

Q: Are test cases independent of each other?
A:

Test cases are not independent of each other. The result of one test case can certainly affect the result of another test case since a test case may change the state of the system or data in the database.

IMPORTANT: You need to carefully think through your test cases and determine how a test case will affect the outcome of subsequent test cases. In fact, you should design your test case sequence so that they put the system in the desired state so that the result of a subsequent test case is what you expect it to be.

Q: During peer acceptance testing, can the tester alter the database or the system time of the computer to do some time related testing? Also can we request the team to be tested to click on previous page, next page, refresh or go to a particular website during testing?
A: The tester cannot directly alter the database or the computer's system time to perform a test case. All testing must be done through the user interface of the website. Clicking on previous, next or refresh is allowed as this is a function of the browser. Going to a third party website or directly to a link of the website being tested without navigating from the front page of the website is not allowed.

Q: Can a team construct a test case that does not do anything except wait for some minutes to see if an event occurs? If it is allowed, it will waste a lot of time so that some remaining test cases cannot be performed.
A:

A tester cannot ask a team to do nothing for some minutes. Not only is the team being tested potentially losing points, but so is the team doing the testing. If a test case requires the passing of time before the result can be checked, then the test case for this should be constructed in such a way that other test cases can be carried out during the waiting period.

Q: Can we test the browser compatibility of a website? Would there be any standard browser for our test cases?
A:

Your system only has to work with IE as installed in Lab 4. This is the only browser it needs to support.

Q: Can we be tested with JavaScript disabled?
A:

If JavaScript is required to be enabled for a system to function correctly, then during the testing session JavaScript must be enabled and it cannot be disabled. In any case, having it not enabled is not considered an error since it is a simple matter for the customer to enable it and for the system to warn the customer that it is not enabled.

Q:

Should the expected result be provided for each test case?

A:

You need to provide the expected result of each test case, otherwise it is not possible to know whether the system passed the test case.
