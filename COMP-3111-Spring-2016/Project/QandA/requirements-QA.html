Q: SRS Requirement: Suppose that a use case involves two actors, A and B, whose interaction with the use case is basically the same. Do we need to write a separate basic flow of this use case for A and B or we should make on of them an alternative flow?
A: You should only write the description once and indicate where and how the flows differ for the different actors. There is no fixed way for how to do this. You just need to make it clear and understandable to everyone who will read your use-case description.

Q: Requirement 1: Is the account number entered by the employee or generated automatically by the system?
A: It should be generated automatically by the system.

Q: Requirement 1: In assigning the number part of an account number, do we need to take into account the first two letters when determining the account number as in the following example: AU00000001, AU00000002, AK00000001, AK00000002 etc.?
A: Yes.

Q: Requirement 1: Can the same client hold several security accounts?
A: No. As far as we can, we restrict each client to hold at most one security account. We see no good reason to allow a client to hold more than one security account.

Q: Requirement 2: You said that we restrict each client to hold at most one account. Is the account you mean the login account or security account.
A: It is the security account.

Q: Requirement 3: What is the valid range of price for each type of security?
A: For bonds and stocks it is 0.00 to 999.99. For unit trusts it is 0.00 to 9999.99.
[Aside: You can determine this yourself by looking at the table definitions in the ExternalDatabases. You can look at the table definitions, you just should not modify them.]

Q: Requirement 4: If a buy or sell order is cancelled, should we send an invoice? Are there any fees in this case?
A: If there is no purchase or sale of securities, then there is no invoice, so nothing needs to be sent and no fees should be charged.

Q: Requirement 4: For a buy or sell order that was executed, should the fees = minimum fee + (%   order type in the table * amount bought or sold)?
A: The fee that should be charged is the max(minimum fee, % order type in the table * amount bought or sold).

Q: Requirement 4: Do we need to check whether an account's remaining balance is sufficient for executing a buy order when the client submits the buy order or does the external system check whether the account's remaining balance is sufficient for execution when it executes a transaction? Also, when a client pays for buying/selling a security, will the external system add/subtract the account's balance automatically when the transaction is executed?
A: The external system has absolutely no access to your system's database, so cannot access a client's account. Allowing this would be a security nightmare! All that the external system does is provide security and currency data to your system and accept and execute orders from your system. Beyond that, everything else is for your system to do.

Q: Requirement 5(c): (1) Can a client cancel a particular alert that he/she set? (2) Should an alert be automatically cancelled if it is triggered?
A: (1) Yes. (2) Yes.

Q: Requirement 5(c): If the client sets an alert on a security and then sells all shares of the security that he holds, should the system delete the alert automatically when the transaction finishes?
A: That would be nice and user friendly and also would meet the requirement that a client can only set an alert on a security that he/she holds.

Q: Requirement 6(c) does not specify whether a user should be able to reorder the listing of the status of any active orders other than by the submission date. Is there any requirement to order the listing by other fields?
A: We have no other requirement for the listing. If your development team would like to allow the listing to be sorted by other fields, that is fine with us, but it is not required.

Q: Requirement 6(d) states that the order history default start and end date should be the current date, but that if no date range is specified than all dates should be included. Does this means that by default only orders from the current date should be displayed and all dates should be included only if the user actually deletes the default dates?
A: Yes.

Posted Apr. 17

Q: Requirement 1: Can an account number start from 0 and be generated randomly?
A: Account numbers should be generated sequentially starting from 1.

Q: Requirement 1: Do we need to handle the case that the client's name contains special characters like '.', '-'?  Do we need to handle the case that a client's name contains non-ASCII characters?
A: There is no restriction on what can be a client's name. Any valid string is allowed that meets the following conditions.
  1. A client's first and last names can be translated or represented in English and relevant, reasonable, visible special characters (including whitespace).
  2. A client's last name contains at least one English letter for generating the account number.

Q: Requirement 4: What is the execution time of an order?
A: An order is considered "executed" when it has either fully completed execution (i.e., its status is "completed") or it has partially executed and its status becomes "cancelled". An order does not have an execution time; only its transactions have an execution time. Taking these two facts into consideration, one can say that the execution time of an order is the execution time of its last transaction.

Q: Requirement 4: What is the total monetary amount of an executed order?
A: The total monetary amount of an executed order is the total amount of all of its transactions (i.e., the sum of (#shares * price) for each transaction). The total monetary amount of an order cannot be determined until the order finishes execution.

Q: Requirement 4: If a client submits a buy order while there are already pending buy orders, should the system regard his "free" cash balance as (balance - total monetary amount of all pending buy orders) when checking whether the balance is sufficient to place the buy order?
A: Requirement 4 states that a client "should be able to place an order to buy securities provided they have sufficient cash in their account". Thus, when a client places a buy order, he/she must have a "free" cash balance sufficient to complete the order. The system should not allow a client to place buy orders whose total monetary value is greater than the client's "free" cash balance at the time that the order is placed. We note that this could result in a client having a negative balance if the total monetary amount of buy orders happens to be larger than what was anticipated when the orders were placed. This situation is acceptable since the overall value of a client's account (i.e., "free" cash balance plus total monetary value of all securities held) should always be greater than zero in the normal case.

Q: Requirement 4: If a client buys a security that causes the account balance to become negative and the security's price subsequently decreases so that the client's assets become negative, is this acceptable?
A: Yes.

Q: Requirement 4: Can we assume that the "order reference number" is assigned sequentially according to the order submission time (i.e., an order submitted later has a larger order reference number)?
A: While order reference numbers are assigned sequentially according to the time an order is received, relying on this fact may not be a good idea as it depends on the external system always assigning reference numbers in this way, which could change and over which you have no control.

Q: Requirement 5(c): Can the high value of an alert be lower than the low value?
A: This should be an error as it makes no sense.

Q: Requirement 2 states that clients cannot change their name unless they send appropriate documents to the company. Does name include the title (i.e. Mr., Mrs., etc.)?
A: Title can be changed by the client.

Q: Requirement 4 states that for calculating the fee for each order "the assets of an account should include the total current value of all security holdings plus the free credit balance". What is the definition of the "current value of all securities". Should it be the value of all securities at the time when the order is detected as completed?
A: Yes.

Q: Requirement 4: Can we place an order to buy or sell 0.01 share of a security?
A: Yes, except for buying stocks.

Q: Requirement 5(a) states that the profit/loss is expected to be shown in 'absolute dollar amount'. How should the profit/loss be shown as an absolute dollar value?
A: When showing profit/loss as a value, there are some standard ways to show a negative value such as with a preceding minus sign or the value enclosed in parentheses () and possibly in red.

Q: Requirement 5(a) states that the previously held securities of a client should be included when calculating the profit/loss. Therefore, should we allow a client to display the profit/loss of an individual security that has already been sold?
A: Yes.

Q: In Requirement 5(c), does "passes through" mean the price change should be from one side to the other side of the alert value to trigger the alert (e.g., if the security price is at 5HKD and a low value alert is then set at 10HKD, will the alert be triggered immediately or is it triggered when the price goes from above 10HKD to below 10HKD)?
A: The value has to actually "pass through" the alert value, either going up for high value alerts or coming down for low value alerts for the alert to trigger. In this case, since the alert is a low value alert, the security value has not "passed through" the alert value (i.e., from above the alert value to below the alert value) so nothing is triggered.

Q: Requirement 5(c): How can the system check the current price of the securities and compare it with the values of the alerts set by clients?
A: You need to program this check.

Posted May 6

Q: Requirement 3: I have found that some of the unit trusts do not have a price value. Is this expected and what is its meaning?
A: It is expected and it means that the unit trust is not yet available for purchase. You still need to show these securities in the search results.

Q: Requirement 3: Is it possible that a client could possess some of the bonds and unit trusts which are not available for purchase (i.e., have no price in the database)?
A: No one can own them because are not yet available for purchase.

Q: In Requirement 4, which currency should we use in the invoice when displaying
  1. total executed dollar amount
  2. fee charged
  3. price per share
A:
  1. total executed dollar amount -> HKD
  2. fee charged -> HKD
  3. price per share -> base currency

Q: In Requirement 4, does "assets" mentioned in the service fee refer to the "balance" of an account or "the total value of securities held plus the balance" of an account?
A: The second one is the usual meaning of some ones assets (i.e., the value of everything that one owns).

Q: For Requirement 4, I have correctly submitted an order (it is stored in the external system), but I still don't have transactions for this order when using getOrderTransaction. Does it take some time for an order to be executed?
A: Orders are executed manually, not automatically. You need to open the ControlCenter.aspx web page inside the ExternalSystems folder and use the "Execute Orders" menu in the navigation bar to retrieve the order and to execute it. Thereafter, you will find any transactions for the order are available to retrieve in your system.

Q: For Requirement 4, if we have placed an order and part of the order is executed, how do we calculate the service fee? Should the service fee be calculated for the whole order or for every transaction?
A: As stated in requirement 4, an invoice is generated after the processing of an order ends. Moreover, an invoice shows the service fee for the order, not for every transaction.

Q: For Requirement 4, if a client submits a stock buy limit order, which price should be used to determine if a client has enough balance to place the order? Should it be
the high price or the current price?
A: It should be the current price.

Q: Requirement 5(a) states that "A client should be able to display the current profit/loss as a percentage." How is the percentage calculated?
A: You may refer to this article: http://www.investopedia.com/ask/answers/05/stockgainsandlosses.asp

Q: Requirement 5(b) states that "A client should be able to show any total monetary value in any of the currencies available from an external system that maintains currency exchange rates." In Activity 1 we were supposed to have completed this requirement. So, where exactly do we need currency conversion support?
A: In Activity 1 you completed the requirement to implement the currency conversion function, but not the requirement to be able to show any total monetary value in any of the currencies available from an external system that maintains currency exchange rates.

You need to provide currency conversion for all the reports in requirement 6 that deal with a client's security holdings that show total monetary value, namely, reports 6(a) and 6(b). It is not required for security searching (requirement 3).

Q: Requirement 5(c): Should we not allow a client to set the low alert higher than the current price?
A: No. Why do you want to prevent this?

Q: Requirement 5(c) When setting high/low alerts, what is the currency of the price that the client enters? Is it HKD or the base currency of the security?
A: It is the base currency of the security.

Q: Requirement 6(a) states that "the submission date of the last executed order" should be shown. How is the term "last executed order" defined?
A: In the usual way. The most recent (last) order that was fully executed according to date and time.
