Requirements Specification

Q:

We are confused about the rules in the project. We made some assumptions about requirements that are not stated in the problem statement. However, there are requirements stated in the Q&A that contradict our assumptions. Do we need to follow the Q&A requirements or continue to follow our assumptions?

A:

In any software development, you should do what the client wants, not what you want! Thus, the first rule of making assumptions is—check your assumptions with the client to see if he/she agrees. We have given you mechanisms to do so, email to 3111rep@cse.ust.hk or Twitter @3111rep, you should use them.

By the way, if you are frustrated about this issue, welcome to the real world! You will get clients/users changing their minds and clarifying requirements all the time. You need to get used to it and learn how to deal with it effectively. You also need to check any assumptions you make with the client. If you do not, you are very likely to build a system the client does not want and what good is that?

Domain Model

(a) Requirements

Q:

 For the domain model, should we just hand in the final domain model or should we hand in the domain model along with the steps that we used to design the domain model, like the list of noun phrases, etc.

A:

 We just want to see the final design of the domain model. The analysis of the problem statement, while useful to you to design the domain model, is not of interest to us.

Q:

 I have some difficulties in understanding what domain model we need to construct for the System Requirements Specification. I have read a book about domain modeling. It mentions that the domain model is a visual representation of the conceptual classes or real-world objects in a domain. It is a conceptual model and it says that responsibilities or methods of a class are not suitable in a domain model. Moreover, in the class examples (ASU: domain model and Video Sales: domain model), we also do not need to include any operations in the domain model. As a result, I am confused about what we need to do for the System Requirements Specification because I think that the domain model required is not the conceptual one and is different from what we learned in class.

A:

First of all, a domain model is represented visually by a class diagram. Different books may use the term domain model to mean different things, but this is what it means in this course. Second, the domain model that you construct for the System Requirements Specification is exactly the same type of model as the domain model for the ASU and Video Sales examples. It is basically the conceptual design of the database for your system. Finally, the domain model classes usually only have operations to get and set the values of the attributes. There is no need to show these operations as it is assumed that every attribute in a domain model class has such operations. Therefore, showing them just clutters the class diagram needlessly.

Q:

What is the scope of the domain model class diagram? Will it represent all the data required by (stored in) the system?

A:

Your domain model class diagram for the System Requirements Specification should contain all the classes and associations required to represent the data requirements of the system.

Q:

Should the domain model represent the database we use in the project? Can we have some class in the domain model for which we do not have a corresponding table in our database?

A:

Since database data are persistent and the domain model represents the persistent data for an application, then the domain model should exactly represent the database data for the project. There should be no class in your domain model that is not represented in some way in your database.

[Technical note: On the other hand, there may well be tables in your implementation that represent the handling of implementation issues and that are not represented in the domain model. This is O.K.]


Q: What should be the level of detail of the domain model UML class diagram? Should it just show the relationships among the classes or does it need to show all the attributes?
A: The detail required is the same as that for the Video Sales domain model example. In addition, it is clearly stated in the System Requirements Specification activity what detail is required in the domain model.

Q: Will the domain model class diagram that we submit be graded? Will we be able to revise it after submission?
A: Yes it will be graded. Yes you can revise it after submission, but the revised domain model will not be graded.
(b) Classes and attributes
Q: Should visibility of the attributes be included in the domain model?
A: No, attribute visibility does not have to be included in the domain model.

Q:

Is it possible to have some classes without any attributes?

A:

Yes. This can happen, for example, when you generalize a class and all of its attributes are inherited from the superclass.

Q:

Is it true that in a domain model two classes cannot share the same attribute?

A:

Correct. Two classes cannot share the same attribute, unless it is inherited through generalization. However, definitely the value of an attribute can never be shared by two objects.

Q:

The lecture notes state that operations should not be specified for domain model classes. What is meant by "operations"? If a user is able to edit some information, is "edit" an operation?

A:

In the lecture notes, "operations" refers to the operations of the class, not the operations that a user can do.

Q:

Should we add some classes in the domain model class diagram to model some temporary data/class (which is required by the system)?

A:

The domain model class diagram contains only persistent data. No temporary data is represented in the domain model class diagram.
(c) Associations

Q:

Can two classes have two different associations between them?

A:

Yes.

Q:

Can we give a name to a binary association that has an association class? I cannot find any example in the notes or even the book where a binary association with an association class has a name.

A:

The name of an association with an association class is the same as the name of the association class.


Use-case Model

(a) Use-case diagram

Q:

For the use case diagram, do I need to show only the final (grouped) use-case diagram or do I need to show more diagrams that show the grouping?

A:

The "grouping" of the functionalities is not part of a use-case diagram. It is only a technique that can help you to decide what should be the final use cases. Only the final use-case diagram needs to be submitted.
(b) Preconditions/postconditions

Q:

I am not very clear about the difference between a constraint and a precondition. It seems to me that they are very similar.

A:

A constraint is a condition that applies all the time in a use case. A precondition is a special case of a constraint. It is a statement of what we expect the state of the system to be before we perform a use case.

Q:

Can you explain more about preconditions and postconditions?

A:

For preconditions you should concentrate on figuring out whether any other use case needs to execute before the use case under consideration can execute. For postconditions, you should probably look mainly for what has to be true in the database or the system after the use case has executed and under what conditions the use case ends. In both cases you are trying to identify any significant states that the system must be in before a use case executes or after a use case finishes. Don't look too hard. Just identify those things that are obvious.

Q:

Can we include a postcondition for every subflow of a use case or we can only include the postcondition for the use case as a whole?

A:

Only for the use case as a whole.
(c) Flow of events

Q:

Do we need to include a part in the use-case model called "Initial step-by-step description", which appears in the lecture notes?

A:

The "Initial step-by-step description" would be done when you are first capturing requirements and want just a high-level overview of the use cases. For the System Requirements Specification, you do not need to do this unless it helps you understand the use cases. As to what to hand in, you only need to hand in your final, refined flow of events for each use case.

Q:

How much detail is required in a use case's flow of events? For example, when describing the processing of invoices we could state

  • The system displays a list of invoices, showing invoice number, shipping account number, ship date, shipment waybill number, service type, package type, weight, sender’s reference number, sender and recipient information, total amount payable and payment due date, and the status according to the selected shipping account.

or we could state

  • The system displays the invoice information.

Which one should we use?

A:

Since a use case description is meant to be a way to communicate clearly with the client about what is required, you need to provide enough detail so that the requirements are clearly stated. Therefore, the first statement is to be preferred because it clearly indicates what is to be displayed.

Q:

Should use-cases include UI specification? I have seen examples in a book that uses both combined, thereby directly describing an actor clicking on a button in the use case, and was wondering which way is the better/proper way to do it.

A:

A use-case description (flow of events) should not refer to the UI. While we know that some books do so, we believe that this is a bad specification technique since it closely ties the use-case (functionality) to the UI. Use-case descriptions should not refer to mouse clicks or dropdown lists or other UI implementation techniques since, if the UI implementation changes, then you also need to update the use-case descriptions. On the other hand, if a use-case description is kept independent of the UI (as the lecture notes state it should be), then you can change the UI without having to update the use-case description provided that the functionality of the use case has not changed.

Q:

 Should we mention exceptional flows in the use-case model? Since the System Requirements Specification handout only mentions a description of the basic flow (normal) of events of the use case, if the use-case model has exceptional flows, what should we do?

A:

We often wonder whether students have read our handouts! It is clearly stated in the System Requirements Specification handout that you are to provide:

"a description of any optional or variant flow of events of the use case (However, DO NOT include exceptional flows);"

While you do need to handle exceptional flows in your implementation, you do not need to describe them in your use-case model.

Q:

In the use-case model we need to include the basic flow of events and any optional or variant flow of events for each use case. If the implementation of our web application changes after the System Requirements Specification is submitted (such that the final implementation is not consistent with the flow of events stated in the System Requirements Specification), will this affect the grading of the System Requirements Specification?

A:

Your final implementation should follow as closely as possible the requirements stated in your System Requirements Specification submission. However, if you find, after you submit your System Requirements Specification, that you missed some requirements, then, of course, you should include the missed requirements in your final system implementation.

Q:

In your examples, you use subflows to describe some operations, such as ADD, DELETE, etc. However, I don't understand what is the difference between:

  1. using subflow to describe an ADD operation and
  2. making another use case ADD?

Which method is better, and why, or are they just the same?

A:

A subflow is used when the functionality is part of the use case. In general one normally would not make a separate use case for each operation. This is too fine a granularity for a use case. You want to have substantial use cases that provide functionality of value to a user.

Q:

I cannot distinguish between an alternative flow and a subflow. What is the difference?

A:

Lets take alternative flow first. An alternative flow is a flow that does not usually happen (e.g., changing a password). These flows happen very rarely.

A subflow is just a way of structuring the main flow of events. So, in the Select Courses to Teach use case, we have two subflows. They are not alternative flows as they are part of the main flow since the user will get the screen allowing him to select one of the three options every time he performs the use case. He will probably not get a screen asking him to change his password every time he performs the use case.

Q:

 Should we put the flow of events in the use-case diagram or in a separate Word file?

A:

 We are not sure what you mean by "in the use-case diagram". For Rational Modeler, do you mean in the Documentation tab for a use case? If so, then yes, you can put it there or you can choose to put it in a separate Word file. Since the formatting capabilities of the Rational Modeler Documentation tab are somewhat limited, our preference is to put it in a Word file so that we can make them look "nice".

Q:

If a use-case diagram has two actors that interact with the same use case at the same time, because they need to do the same use-case together, how do I write the flow of events? Do I write one flow of events for both actors or do I write two flow of events, one for each actor?

A:

 A use-case description is not written for each actor that uses a use case! It is written for the use case. If the actors that can use the use case can do different things, you should possibly consider having two use cases. If they can do exactly the same thing, then the fact that there are two different actors that use this use case should not affect how you write the flow of events. If there are some parts of the flow of events that are actor specific, then you can say something like: "If the actor is X then ...". Such actor-specific parts in a flow of events should be few. If they are many and extensive, then it may be better to have a separate use case for each actor.

Q:

Is it the case that for each actor/use case pair there will be one flow of events? For example, if a staff actor is involved in a use case and a customer actor is also involved in the same use case, is there one flow of events describing the use case from the view of the customer and one from the view of the staff?

A:

There is only one flow of events for each use case, no matter how many actors it may have. The flow of events is for the use case, not the actor. However, there may be some actor specific parts in a use case that only one of the actors executes. In such a case, you can include an actor specific test in the use case to determine if that functionality should be executed as described in the previous question. There is obviously a design trade-off here. If a use case has too many actor specific parts, then maybe it is better to have separate use cases for each of the actors.

Q:

Can we replace all "non-error" alternative flows by using if-statements in the basic flow as stated in the "System Requirements Capture" lecture notes?

A:

Yes. While there is no fixed way or requirement for how you write a flow of events, it should be clear and tell the whole story of the use case. In this regard, using too many if-statements in a flow of events can obscure the basic flow so that it is not obvious or easily understandable. Thus, extensive use of if-statements to show alternative flows is not recommended. For clarity and ease of understanding, it is usually better to put them in a separate alternative flows section of a use-case.

Q:

I have grouped some small use cases into a big use case. However, for each small use case, they have their own alternative flows. How can I represent all the alternative flows in one big use case diagram which has grouped the use cases?

A:

First of all, you do not use use cases as you would procedures and functions. In particular, you do not "group" use cases into larger use cases. Each use case should stand on its own and tell a complete "story" about one of the system's functionalities.

Second, you do not show alternative flows in the context use-case diagram that shows all the actors and all the major use cases that provide the system's functionality. Your context use-case diagram should be simple and show only the major (main) use cases.

[NOTE: The idea of the context use-case diagram is to show the system's major functionality and to be simple to understand. If it becomes so complex that you cannot even draw it, then you probably have a bad design for your use-case model!]
(d) Nonfunctional requirements

Q:

What is a nonfunctional requirement?

A:

A nonfunctional requirement is a requirement that does not provide any benefit directly to the user in terms of what they can do using the system. An example of a nonfunctional requirement might be that the system (or a particular use case) be implemented in a certain programming language. From the users' perspective this makes no difference to the functionality provided by the system, but it places a constraint on the implementation of the system. Please see sections 4.3.2 and 4.4.7 of the text book for more details. Also, doing a Google search, we found the following very good page on nonfunctional requirements: http://www.csee.umbc.edu/courses/undergraduate/345/spring04/mitchell/nfr.html.
(e) Administration use cases

Q:

The System Requirements Specification handout states that the use-case model should not include any "administration use case." Can the actor for an "administration use case" be a human?

A:

Yes.

Q:

Would you explain more about the term "administration use case?" Does it refer to all the use cases that are not end user oriented?

A:

Basically, yes. There are some examples of administration use cases listed in the System Requirements Capture lecture notes. If you stick to capturing and representing in use cases only the requirements stated in the problem statement and the Q&A, then you should be O.K.

Q:

In the System Requirements Specification handout, we are asked not to include any administration use case. Does administration use case mean "setting up of new accounts," "report generation," etc.? Since the reports generated can be viewed by the users, it seems that such a use case can provide functionality to the users.

A:

Every requirement specified in the course project problem statement and the Q&A is user related. Therefore, if the problem statement or the Q&A includes a requirement to set up new accounts or generate reports, then it needs to be included in the System Requirements Specification.
(f) External systems

Q:

 Do we need to include external systems in the use-case model?

A:

 If the system you are designing and implementing interacts with an external system (gets data from it or sends data to it) then it needs to be included in the use-case model.

Q:

 Do we need to write use cases for an external system? The System Requirements Specification handout states that "The use-case model should include only user-level use cases ... ". However, if we do not need to write these use cases, then do we need to write acceptance tests for the external system?

A:

Use cases are only written for the system being designed to describe what happens inside the system. Actors (which include both humans and external systems) are outside the system. They are only described briefly. Since they are outside the system being designed, you do not know exactly what happens inside them so you cannot describe them using use cases. Furthermore, acceptance tests are also only for the system being designed. You have no control over or means to test an external system, just like you have no way to test a human actor.
